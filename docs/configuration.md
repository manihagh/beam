# Configuration reference

Everything is env-var driven via `wrangler.toml`. The same code runs for any company; you just configure it.

## Required values

These have no sensible defaults and must be set.

| Variable | Where | Description |
|---|---|---|
| `routes` | `wrangler.toml` | Three custom domains: internal control plane, MCP host, public share host. |
| `SERVE_HOST` | `[vars]` | Hostname of the internal control plane. Must match one of the routes. |
| `MCP_HOST` | `[vars]` | Hostname of the MCP endpoint. Must match one of the routes. |
| `PUBLIC_HOST` | `[vars]` | Hostname of the public share viewer. Must match one of the routes. |
| `ALLOWED_SSO_DOMAINS` | `[vars]` | Comma-separated domains your staff use. Defense-in-depth check on top of Cloudflare Access. |
| `ADMIN_EMAILS` | `[vars]` | Comma-separated admin email list. Admins can issue permanent share links and override owner-only mutations. |
| `BRAND_FROM_EMAIL` | `[vars]` | The address transactional emails appear from. |
| `bucket_name` | `[[r2_buckets]]` | R2 bucket name. The deploy script will create it. |

## Branding

| Variable | Default | Description |
|---|---|---|
| `BRAND_NAME` | `"Beam"` | Display name in the control plane, emails, and MCP `serverInfo`. |
| `BRAND_PRIMARY_COLOR` | `"#B5482A"` | Hex color used as the accent throughout. Validated; falls back to default if invalid. |

## Share link policy

| Variable | Default | Range |
|---|---|---|
| `DEFAULT_SHARE_HOURS` | `72` | 1 to 168 |
| `MIN_SHARE_HOURS` | `1` | 1 to 168 |
| `MAX_SHARE_HOURS` | `168` (1 week) | 1 to 168 |

The hard upper bound is 168 hours (1 week) for share links issued via Claude or the API. Permanent share links require an admin and are issued only via the control plane UI.

## Versioning

| Variable | Default | Range |
|---|---|---|
| `DASHBOARD_VERSIONS_KEPT` | `10` | 0 to 50 |

Number of historical versions of each dashboard's HTML to retain in R2. Set to `0` to disable versioning (`update_dashboard` overwrites in place).

## Secrets

Set these via `wrangler secret put NAME` (not `[vars]`). Storing them as vars is wrong; Cloudflare prints vars in the dashboard.

| Secret | Required | Description |
|---|---|---|
| `MCP_BEARER_TOKEN` | yes (auto-generated by deploy.sh) | Local-dev fallback auth and HMAC seed for password-challenge cookies. |
| `GMAIL_CLIENT_ID` | no | Gmail OAuth client id for outbound email. |
| `GMAIL_CLIENT_SECRET` | no | Gmail OAuth client secret. |
| `GMAIL_REFRESH_TOKEN` | no | Gmail OAuth refresh token, bound to a single workspace user. Scope `gmail.send` only. |
| `UNSUB_SECRET` | no | 32+ char random string used to sign unsubscribe tokens. |

If any Gmail secret is missing, email sends are silently skipped. The system still works; recipients just do not get publish/share/visibility notifications.

## Adding a new admin

Edit the `ADMIN_EMAILS` value in `wrangler.toml`, then re-run `./deploy.sh`. There is no separate admin database; the env var is the source of truth.

## Adding a new SSO domain (e.g., after an acquisition)

1. Add the domain to `ALLOWED_SSO_DOMAINS` in `wrangler.toml`.
2. Update the Cloudflare Access policy on `beam.YOUR_COMPANY.com` and `mcp-beam.YOUR_COMPANY.com` to allow the new domain.
3. Re-run `./deploy.sh`.

Both changes are required. Forgetting step 2 means new-domain users get rejected at Access. Forgetting step 1 means they get past Access but the Worker rejects them with `sso_domain_not_allowed`.

## Local development

In addition to the production vars, `.dev.vars` (gitignored) sets:

```
MCP_BEARER_TOKEN=dev-test-token
```

`wrangler dev` reads this for local-only auth. Real production uses Cloudflare Access on the MCP host.

## Validation

The Worker validates configuration on every request via `getConfig(env)`. Invalid values fall back silently:

- Invalid `BRAND_PRIMARY_COLOR` → falls back to default indigo.
- `DEFAULT_SHARE_HOURS` outside 1-168 → clamped to range.
- `DASHBOARD_VERSIONS_KEPT` above 50 → clamped to 50.
- Missing `BRAND_FROM_EMAIL` → derived as `noreply@{SERVE_HOST}`.

This means a misconfiguration will not break the deployment; it will just behave like the default. Check `/whoami` on the control plane to verify the resolved config.
